# Movie Maker Roadmap & Test Plan

本文件描述当前项目距离”类似 Windows Movie Maker 的视频剪辑软件”还差哪些能力，并给出分阶段的开发与测试计划。

## 当前状态（已实现）

### 基础架构 ✅
- 构建与测试：`npm run build` 可通过；`npm test` 全绿。
- 开发效率：`npm run dev:all` 一键启动（前端+调度器）；`npm run dev:restart` 清理并重启。
- 状态管理：Zustand store 支持 tracks/scenes/assets/keyframes 数据模型。
- 持久化：工程 JSON 保存/加载（version=1）。

### 时间轴交互 ✅
- 素材拖拽落点、吸附预览（frame/second）。
- 播放头 scrub、播放时自动滚动跟随。
- 片段鼠标拖动、位置更新。
- 分割场景（splitScene）、复制场景（duplicateScene）。
- 多轨支持：video/audio/text 轨道分离。

### 音频体验 ✅
- 音频轨视觉（波形）、防重叠自动避让。
- 添加视频素材时自动创建对应音频片段。
- 轨道音量控制（volume）。

### 预览 ✅
- Remotion composition 根据 `tracks/assets` 渲染 `Img/Video/Audio`。
- Player 事件与 store 同步（currentFrame/isPlaying）。
- 播放到末尾停止。

### 素材管理 ✅
- 素材上传（image/video/audio）。
- 素材列表、筛选（assetFilter）。
- 素材删除、选择。

### 差距总结

**🔴 严重差距（阻碍基本使用）**
1. **片段裁剪（Trim）**：无法调整片段的入点/出点，这是视频编辑最基础的功能。
2. **素材持久化**：blob: URL 跨会话失效，工程文件无法复用。
3. **转场效果**：无任何转场，片段之间生硬切换。
4. **撤销/重做**：误操作无法恢复，用户体验极差。
5. **导出流程**：无完整的导出 UI 和进度反馈。

**🟡 中等差距（影响专业使用）**
1. **多轨编辑**：仅单视频轨/单音频轨，无法实现画中画、多音轨混音。
2. **音频精细控制**：无片段级音量包络线、淡入淡出。
3. **文本/字幕**：有数据结构但无编辑 UI。
4. **特效系统**：无视频特效（亮度/对比度/色调/滤镜）。

**🟠 较低差距（影响效率）**
1. **快捷键系统**：无完整快捷键支持。
2. **时间轴缩放**：有状态但无 UI 控制。
3. **轨道高度调整**：固定高度，无法适应不同需求。

## 里程碑计划

时间仅为建议（按 1 人开发估算）。每个里程碑都给出验收标准。

### Milestone 0：稳定性收敛 ✅ 已完成

**目标**：把”现在已有的”变得稳定、可持续迭代。

**范围**：
- ✅ 工程加载后 `tracks/scenes/assets` 一致性校验与修复（避免漂移/重复）。
- ✅ 缺失素材处理：被引用素材缺失时预览不崩溃、时间轴有明显提示（红色边框+图标）。
- ✅ 波形解码：并发限制（最多 3 个）、失败回退（显示占位符）、缓存上限（100MB）与释放策略。
- ✅ 素材元数据提取：视频时长/分辨率、音频采样率、图片尺寸（使用 Web API）。
- ✅ 错误边界：预览崩溃时不影响编辑器，显示友好错误提示。

**验收标准**：
- ✅ 10 分钟连续拖拽/播放/缩放不崩溃、不明显卡顿（CPU < 80%）。
- ✅ 缺失素材工程能打开、时间轴有明显标记、导出前给明确提示并阻止。
- ✅ 加载包含 20+ 素材的工程，波形解码不阻塞 UI。

**实现细节**：
- 创建了 `src/lib/metadata-extractor.ts` 工具模块，支持视频/音频/图片元数据提取。
- 更新了 `AssetUploader` 组件，在上传时自动提取元数据。
- 扩展了数据库 schema，添加 `sample_rate` 和 `number_of_channels` 字段。
- 更新了 `Asset` 接口，支持音频元数据字段。
- 添加了完整的单元测试（10 个测试用例全部通过）。
- 所有集成测试通过（123 个测试用例）。

### Milestone 1：可用编辑 MVP（1.5–2 周）🎯 核心目标

**目标**：完成”导入→剪辑→预览一致→导出”的最小闭环，达到 Windows Movie Maker 基础可用水平。

**范围**：

#### 1.1 片段裁剪（Trim）- 3 天
- 左右手柄拖拽裁剪（调整 in/out 点）。
- 手柄 hover 高亮、拖拽时显示裁剪时长。
- 裁剪时保持片段位置不变（只改变 durationFrames）。
- 最小片段时长限制（1 秒 = 30 帧）。

#### 1.2 素材持久化 - 2 天
- 将 `blob:` URL 升级为持久化存储：
  - 方案 A：`/public/uploads/{projectId}/{assetId}.{ext}`（简单，适合单机）。
  - 方案 B：IndexedDB（适合浏览器环境，无服务器依赖）。
- 工程 JSON 记录相对路径与元数据（duration/width/height/sampleRate）。
- 加载工程时检查素材文件是否存在，缺失时标记为 `missing: true`。

#### 1.3 导出完整流程 - 3 天
- 导出面板 UI：格式选择（MP4/WebM）、分辨率、帧率、质量。
- 导出前检查：缺失素材、空时间轴、音频轨冲突。
- 导出进度条：显示当前帧/总帧数、预计剩余时间。
- 导出完成：下载文件、显示成功提示。
- 使用 Remotion `renderMedia` API 或 `@remotion/lambda`（云端渲染）。

#### 1.4 撤销/重做 - 2 天
- 实现 undo/redo 栈（最多保留 50 步）。
- 覆盖操作：addScene、updateScene、deleteScene、splitScene、updateScenePosition。
- 快捷键：Cmd/Ctrl+Z（撤销）、Cmd/Ctrl+Shift+Z（重做）。
- UI 指示：撤销/重做按钮状态（禁用/启用）。

#### 1.5 多轨编辑增强 - 2 天
- 支持多条视频轨（至少 3 条）、多条音频轨（至少 3 条）。
- 轨道添加/删除 UI（右键菜单或工具栏按钮）。
- 轨道高度拖拽调整（最小 60px，最大 200px）。
- 轨道重命名、显示/隐藏、锁定/解锁。

**验收标准**：
- ✅ 可完成一个短片：导入 mp4 + 配音 + 图片 → 排时间轴 → trim/分割 → 预览一致 → 导出 mp4（含音频）。
- ✅ 误操作后可撤销恢复，不丢失数据。
- ✅ 导出的视频与预览完全一致（时长、音画同步、特效）。
- ✅ 工程保存后关闭浏览器，重新打开可继续编辑（素材不丢失）。

### Milestone 2：接近 Movie Maker 完整体验（2–3 周）🚀 产品化

**目标**：提升编辑效率与表达能力，达到 Windows Movie Maker 的完整功能水平。

**范围**：

- 转场预览：拖拽到片段之间自动创建转场对象。
- 转场库面板：缩略图预览、拖拽应用。

#### 2.2 文本/字幕系统 - 4 天

- 文本编辑器：
  - 基础样式：字体（5 种）、大小（12–72px）、颜色、粗细。
  - 高级样式：描边、阴影、背景色、对齐方式。
  - 动画预设：淡入、飞入、打字机效果（3 种）。

#### 2.3 音频精细控制 - 3 天
- 片段级音量调节：音量包络线（keyframe）。
- 淡入淡出：拖拽片段边缘创建淡化效果（0.5–2 秒）。
- 音频波形优化：缓存预计算波形数据到 IndexedDB。
- 音频特效预设：降噪、均衡器（低音/高音）、压缩器（3 种预设）。

#### 2.4 快捷键系统 - 2 天
- 播放控制：Space（播放/暂停）、Home/End（跳到开头/结尾）。
- 编辑操作：Delete（删除）、Cmd/Ctrl+D（复制）、Cmd/Ctrl+X（剪切）。
- 时间轴导航：←/→（逐帧）、Shift+←/→（逐秒）、+/−（缩放）。

#### 2.5 时间轴体验优化 - 3 天
- 时间刻度尺：清晰显示秒/分钟标记、当前时间。
- 缩放控制：滚轮缩放、缩放滑块（0.1x–5x）。
- 虚拟化渲染：只渲染可视区域的片段（支持 1000+ 片段）。
- 吸附增强：吸附到片段边缘、播放头、关键帧（可视化吸附线）。

#### 2.6 视频特效（基础）- 3 天
- 实现 5 种基础特效：
  - 亮度/对比度
  - 饱和度/色调
  - 模糊
  - 锐化
  - 黑白/复古滤镜
- 特效面板：滑块调节参数、实时预览。
- 特效叠加：支持多个特效同时应用。

**验收标准**：
- ✅ 5 分钟内制作”带字幕+转场+配乐+特效”的 30 秒视频，不需要刷新页面。
- ✅ 快捷键流畅操作，不需要频繁使用鼠标。
- ✅ 时间轴缩放到 100+ 片段时仍然流畅（60fps）。
- ✅ 导出视频包含所有转场、字幕、特效。

```typescript
describe('Undo/Redo Integration', () => {
  it('should undo scene addition', () => {
```

### 端到端测试（Playwright）⚠️ 需要新增

**目标**：真实用户操作链路稳定。

**优先级 P0（Milestone 1 必须）**：
1. **基础编辑流程**：
   - 导入素材（图片/视频/音频）→ 拖拽到时间轴 → 播放预览 → 导出视频。
   - 验证：导出的视频文件存在、时长正确、可播放。

2. **Trim 操作**：
   - 拖拽左手柄裁剪 → 预览更新 → 导出验证裁剪生效。
   - 拖拽右手柄裁剪 → 预览更新 → 导出验证裁剪生效。

3. **撤销/重做**：

### 性能/稳定性测试

**目标**：确保大型工程下的流畅体验。

**测试场景**：

1. **大型时间轴（200+ 片段）**：
   - 生成 200 个片段、总时长 20 分钟的工程。
   - 测试指标：
     - 滚动时间轴：60fps 不掉帧。
     - 拖拽片段：延迟 < 50ms。
     - 播放预览：CPU < 80%、内存 < 2GB。
   - 工具：Chrome DevTools Performance、React Profiler。

2. **长时间播放（10 分钟连续播放）**：
   - 播放 10 分钟视频，验证：
     - `currentFrame` 与播放头位置不漂移（误差 < 1 帧）。
     - 音画同步（误差 < 100ms）。
     - 内存不泄漏（增长 < 100MB）。

3. **波形解码压力测试**：
   - 同时导入 50 个音频文件。
   - 验证：
     - 并发解码数量 ≤ 3。
     - UI 不阻塞（可继续编辑）。
     - 缓存命中率 > 90%（重复加载时）。

4. **导出压力测试**：
   - 导出 5 分钟、1080p、30fps 视频。
   - 验证：
     - 导出时间 < 10 分钟（2x 实时速度）。
     - 导出过程中可继续编辑（不阻塞 UI）。
     - 导出失败时有明确错误提示。

**性能基准**：
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 时间轴滚动 FPS | ≥ 60 | Chrome DevTools Performance |
| 片段拖拽延迟 | < 50ms | Performance.now() |
| 播放 CPU 占用 | < 80% | Activity Monitor / Task Manager |
| 内存占用（200 片段） | < 2GB | Chrome DevTools Memory |
| 波形解码并发数 | ≤ 3 | 日志统计 |
| 导出速度 | ≥ 2x 实时 | 时间统计 |

## 风险与建议

### 技术风险

1. **素材持久化方案选择** 🔴 高风险
   - **风险**：`blob:` URL 跨会话失效，工程文件无法复用。
   - **方案对比**：
     - 方案 A：文件系统（`/public/uploads`）- 简单但需要服务器支持。
     - 方案 B：IndexedDB - 纯浏览器方案，但有存储限制（通常 50MB-1GB）。
     - 方案 C：云存储（S3/OSS）- 最稳定但需要后端和成本。
   - **建议**：Milestone 1 先用 IndexedDB（快速验证），Milestone 2 升级到云存储。

2. **导出性能瓶颈** 🟡 中风险
   - **风险**：Remotion 导出依赖 Node.js 环境，浏览器端导出性能差。
   - **方案对比**：
     - 方案 A：浏览器端 `renderMedia`（慢，5 分钟视频需 20+ 分钟）。
     - 方案 B：`@remotion/lambda`（快，但需 AWS 账号和费用）。
     - 方案 C：自建渲染服务器（灵活，但需要运维）。
   - **建议**：Milestone 1 先用浏览器端（验证流程），Milestone 2 接入云端渲染。

3. **大型时间轴性能** 🟡 中风险
   - **风险**：200+ 片段时滚动/拖拽卡顿。
   - **解决方案**：
     - 虚拟化渲染（react-window/react-virtualized）。
     - 节流/防抖（拖拽、滚动事件）。
     - Web Worker 处理波形解码。
   - **建议**：Milestone 1 先支持 50 片段，Milestone 2 优化到 200+。

4. **音画同步精度** 🟠 低风险
   - **风险**：长时间播放时音画不同步（累积误差）。
   - **解决方案**：
     - 使用 Remotion 的 `useCurrentFrame()` 确保帧精确。
     - 定期校准播放头位置（每秒检查一次）。
   - **建议**：Milestone 1 先支持 5 分钟视频，Milestone 2 优化到 30 分钟。

### 产品风险

1. **用户学习成本** 🟡 中风险
   - **风险**：功能过于复杂，用户不会用。
   - **解决方案**：
     - 新手引导（首次使用时的交互式教程）。
     - 示例工程（预置 3 个模板工程）。
     - 快捷键提示（Hover 时显示）。
   - **建议**：Milestone 2 加入新手引导和示例工程。

2. **浏览器兼容性** 🟠 低风险
   - **风险**：Safari/Firefox 兼容性问题。
   - **解决方案**：
     - 优先支持 Chrome（90% 用户）。
     - 使用 Polyfill（Web Audio API、File System API）。
     - 明确标注浏览器要求（Chrome 90+）。
   - **建议**：Milestone 1 只支持 Chrome，Milestone 3 扩展到其他浏览器。

### 开发建议

1. **优先级排序原则**：
   - 🔴 P0：阻碍基本使用的功能（Trim、素材持久化、导出）。
   - 🟡 P1：提升用户体验的功能（撤销/重做、转场、字幕）。
   - 🟠 P2：锦上添花的功能（高级特效、协作、云端渲染）。

2. **迭代策略**：
   - **快速验证**：Milestone 1 用最简单方案验证核心流程（2 周）。
   - **逐步优化**：Milestone 2 优化体验和性能（3 周）。
   - **持续迭代**：Milestone 3 根据用户反馈持续改进（长期）。

3. **技术债务管理**：
   - 每个 Milestone 结束后预留 2 天重构时间。
   - 代码审查：关键模块（store、timeline、export）必须 review。
   - 文档更新：API 变更及时更新文档。
